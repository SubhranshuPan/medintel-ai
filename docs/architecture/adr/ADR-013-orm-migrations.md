# ADR-013 — SQLAlchemy 2.0 (async) + Alembic for ORM and Migrations

## Status

Accepted

## Context

Sprint 1 (Backend Foundation, issue #6) introduces the relational data layer for the core clinical entities (User, Conversation, Message, Document, Embedding, Citation). The backend is FastAPI-based and asynchronous (ADR-001) over PostgreSQL (ADR-003), and the project follows a repository/service pattern that keeps persistence concerns out of the API and business layers.

This requires two decisions that were not yet captured as an ADR: which ORM to use, and how to manage schema migrations. Both are foundational and hard to reverse once models and migration history exist, so they are recorded here.

## Decision

**SQLAlchemy 2.0 (async) is adopted as the ORM, and Alembic as the migration tool.**

- **Async engine** via the `asyncpg` driver (`postgresql+asyncpg://…`). A single async engine and `async_sessionmaker` live in `app/core/db.py`; a `get_db()` dependency yields a request-scoped session that commits on success and rolls back on error.
- **Typed models** using SQLAlchemy 2.0 `Mapped[...]` / `mapped_column(...)` declarative style. Shared `UUIDMixin` (UUID primary keys — non-enumerable, avoids leaking record counts or guessable patient identifiers) and `TimestampMixin` (`created_at` / `updated_at`, DB-maintained) apply to every table for GDPR-aware auditability.
- **Repository pattern**: a generic `BaseRepository[ModelT]` (`app/repositories/base.py`) is the single choke point for ORM access; services and API handlers do not touch the session directly.
- **Alembic in async mode** (`alembic init -t async`). The migration URL is sourced from application settings (`MEDINTEL_DATABASE_URL`) rather than hard-coded in `alembic.ini`, keeping credentials out of the repository. Autogenerate runs with `compare_type=True`.
- **Enum handling**: PostgreSQL `ENUM` types (`user_role`, `message_role`) are created implicitly by `create_table` on upgrade; downgrades explicitly drop them so the migration cycle is fully reversible and idempotent (verified with upgrade → downgrade → upgrade against a live Postgres).

## Alternatives Considered

- **SQLModel** — thin wrapper over SQLAlchemy + Pydantic; less control over the mapping layer and still maturing for complex async use; keeping SQLAlchemy direct avoids a leaky abstraction.
- **Tortoise ORM / Piccolo** — async-native, but smaller ecosystems and weaker migration tooling than Alembic; less recognised by interviewers.
- **Raw `asyncpg` + hand-written SQL** — maximal control, but forfeits typed models, relationship management, and autogenerated migrations; more error-prone for a schema that will grow.
- **Django ORM / Prisma** — would pull in a different framework or a Node toolchain, conflicting with the FastAPI modular-monolith stack (ADR-001, ADR-008).

## Consequences

### Positive

- SQLAlchemy 2.0 + Alembic is the de facto standard for production async Python APIs and is widely recognised by interviewers — directly supporting the portfolio goal.
- Typed `Mapped` models give editor/type-checker support and self-documenting schemas.
- The repository pattern isolates persistence, keeping the API layer testable and the ORM swappable in principle.
- Alembic autogenerate + `alembic check` give a drift-detectable, reversible migration history from day one.

### Negative

- Async SQLAlchemy has sharper edges than sync (greenlet bridging, `run_sync` in Alembic, no lazy-loading in async context) — a small learning-curve cost.
- Alembic autogenerate does not handle PostgreSQL ENUM drops on downgrade automatically; migrations touching enums need a manual drop (as done in the initial migration). This is a known, documented gotcha rather than a one-off.
- Two code paths conceptually exist for enums (implicit create, explicit drop); future enum changes must remember the downgrade side.
